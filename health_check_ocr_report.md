# 건강검진(health_check) OCR 파싱 로직 수정 및 검증 결과 보고서

작성: 신재욱 (브랜치 `feature/재욱-ocr-eval-compare`) · 작성일: 2026-06-08

---

## 1. 작업 배경

`health_check_*.pdf`(공단 양식) 추출 정확도가 65.4%로 낮게 나와, 좌표 기반 추출(`pdfplumber.extract_words`)과 파싱 로직을 점검·수정함. 작업 순서는 다음과 같음.

1. 부정확하게 나오는 항목(혈압, 키/몸무게/BMI) 원인 분석
2. `extractor.py` / `pdf_handler.py` 코드 수정 및 5건 단위 검증
3. 1,000건 규모 검증으로 수정 효과 확인
4. 10,000건 전체 검증 실행 및 결과 분석 → 데이터셋 자체의 결함 발견

---

## 2. 코드 수정 내역

### 2-1. 키/몸무게/BMI 교차 오염 수정 (`extractor.py` · `parse_height_weight`)

**원인**: 키 키워드 주변 15줄에서 숫자를 모두 후보로 모으는 과정에서, 같은 영역에 있는 BMI 수치(예: 21.2)가 "키보다 작은 첫 번째 숫자" 조건에 맞아 몸무게로 잘못 추출되는 경우가 있었음. 이렇게 잘못 추출된 몸무게 값이 이후 BMI 교차검증 로직에 들어가면서, 원래 정확히 추출되어 있던 BMI 값까지 덮어써 틀리게 만드는 연쇄 오류로 이어짐.

**수정**: 후보 숫자를 모으는 단계에서 BMI 키워드가 포함된 줄은 제외하도록 분기를 추가.

```python
def parse_height_weight(text_lines):
    for i, (text, _) in enumerate(text_lines):
        if is_keyword_match(text, FIELD_KEYWORDS["height_cm"]):
            context = text_lines[i : i + 15]
            all_nums = []
            for ctx_text, _ in context:
                # BMI 줄의 수치는 몸무게 후보로 잘못 집힐 수 있으므로 제외  ← 신규 추가
                if is_keyword_match(ctx_text, FIELD_KEYWORDS["bmi"]):
                    continue
                all_nums.extend(extract_numbers(ctx_text))
            ...
```

**기대 효과**: BMI 값이 몸무게 후보 풀에서 원천 제거되므로 몸무게가 정확해지고, 그 결과 BMI 교차검증도 더 이상 올바른 값을 잘못 덮어쓰지 않게 됨.

### 2-2. 혈압(수축기/이완기) 추출 오류 수정 (`pdf_handler.py` · `_group_values_prefer_synthetic`)

**원인 분석 과정**: 처음에는 "x0 ≤ 460 컷오프가 실제 혈압 수치를 잘라낸다"는 가설을 세웠으나, 좌표 덤프 스크립트(`_debug_bp_coords.py`)로 직접 확인한 결과 실제 혈압 수치는 `x0 ≈ 166~191` 구간에 위치해 460 컷오프와 무관함이 확인되어 이 가설은 기각함.

실제 원인은 "합성 레이어(같은 자리에 중복 인쇄된 텍스트) 제거" 로직에 있었음. 이완기 혈압 수치(예: 64)의 `x0(191.1)`와 구분자 "/"의 `x0(185.5)` 사이 거리가 5.6pt에 불과한데, 기존 중복 판정 허용 오차(`x_tolerance=8.0`)가 이보다 커서 둘을 "같은 값의 중복 레이어"로 오판하고, 그중 "/"만 남기고 실제 수치(64)를 버려버리는 문제였음. 그 결과 추출된 원문 텍스트 어디에도 이완기 혈압 수치가 존재하지 않게 됨.

**수정**: 중복 판정 허용 오차를 `8.0 → 3.0`으로 축소.

```python
def _group_values_prefer_synthetic(words, top_tolerance=3.0, x_tolerance=3.0):
    # 변경 전: x_tolerance=8.0
```

실제 동일 레이어의 중복 수치는 x0 차이가 거의 0에 가깝고, 이번에 발견된 최소 실제 간격은 약 5.6pt였으므로, 허용 오차를 3.0으로 낮추면 진짜 중복은 그대로 제거하면서 인접한 서로 다른 수치(혈압의 수축기/이완기 등)는 보존할 수 있음.

**검증**: 수정 후 `_debug_health_check_raw.py`로 재확인한 결과, 테스트 표본에서 수축기/이완기 혈압, 키, 몸무게, BMI 5개 항목 모두 정답과 100% 일치함을 확인함.

## 3. 1,000건 규모 검증 결과

수정한 코드를 반영해 health_check 표본 1,000건(정상 구간)을 재평가함.

| 항목 | 결과 |
|---|---|
| 전체 평균 정확도 | **99.7%** |
| 혈압(수축기/이완기), 키, 몸무게, BMI, 공복혈당 등 핵심 항목 | **모두 99.5% 이상** |

→ 2-1, 2-2에서 수정한 두 가지 파싱 로직이 실제로 효과가 있음을 확인했고, "파싱 규칙 자체의 문제"는 해소된 것으로 판단함.

## 4. 10,000건 전체 검증에서 발견된 문제 — "특정 구간부터 정확도가 급락"

표본 1,000건에서는 99%대였던 정확도가, 10,000건 전체로 확대하자 **평균 66.1%**로 급락함. 원인을 규명하기 위해 실패 상세 데이터(필드별 GT vs OCR 값)를 JSON으로 남기도록 검증 스크립트를 개선하고 실패 패턴을 분석함.

### 4-1. 1차 단서: 특정 오답 값이 수천 건에서 "고착"

서로 다른 피험자(GT 값이 모두 다름)의 파일에서 동일한 OCR 오답 값이 수천 건 반복되는 비정상 패턴을 발견함.

| 필드 | 고착된 OCR 값 | 해당 값으로 실패한 건수 |
|---|---|---|
| 수축기 혈압 (systolic_bp) | 114.0 | 8,293건 |
| 이완기 혈압 (diastolic_bp) | 79.0 | 8,222건 |
| 키 (height_cm) | 166.7 | 1,000건 |
| 몸무게 (weight_kg) | 54.0 | 1,001건 |
| 체질량지수 (bmi) | 19.4 | 1,000건 |
| 공복혈당 (fasting_glucose) | 82.0 | 3,340건 |

처음에는 "PaddleOCR 엔진/파이프라인이 대량 순차 처리 중 메모리 누수나 캐시 오염으로 같은 결과를 계속 재사용하는 버그"로 의심했음. 그러나 실패 파일 목록을 순서대로 살펴보니, **6개 항목 모두 예외 없이 정확히 `health_check_01001.pdf`부터** 고착 현상이 시작되고 00001~01000번까지는 정상이었음. 점진적으로 나빠지는 누수형 패턴이 아니라 "1000건 단위에서 칼같이 끊기는" 구조적 경계였다는 점에서 "파이프라인이 망가진다"는 가설과는 결이 다르다고 판단해, 추가 검증 스크립트(`_debug_stuck_value.py`)로 실제 PDF 원문을 직접 열어 확인함.

### 4-2. 최종 원인 확인: OCR이 아니라 "PDF 원본 데이터 자체의 결함"

`health_check_01000.pdf`(정상 판정)와 `health_check_01001 / 01002.pdf`(고착 시작)의 PDF 원문 텍스트를 직접 추출해 비교함.

| 파일 | 피험자(GT) 정답값<br>(수축기/이완기/키/몸무게/BMI/혈당) | PDF에 실제로 인쇄된 값 |
|---|---|---|
| health_check_01001.pdf | 143 / 73 / 145 / 45 / 21.4 / 99 | 114 / 79 / 166.7 / 54.0 / 19.4 / 82 |
| health_check_01002.pdf | 119 / 75 / 165 / 60 / 22.0 / 91 | 114 / 79 / 166.7 / 54.0 / 19.4 / 82 |

두 파일은 서로 다른 피험자(`sample_01001`, `sample_01002`)이고 정답(GT)도 서로 다른데, **PDF 안에 인쇄되어 있는 값은 완전히 동일함**. 그리고 우리 OCR이 추출한 값(114, 79, 166.7, 54.0, 19.4, 82)은 이 "PDF에 인쇄된 값"과 정확히 일치함.

> 즉, OCR/파싱 파이프라인은 PDF에 적힌 내용을 한 글자도 틀리지 않고 정확하게 읽어낸 것이고, 문제는 **"PDF 생성 단계에서 1001번째 피험자부터 서로 다른 사람의 PDF에 동일한 값이 인쇄된 것"**, 즉 평가용 데이터셋(health_check PDF + `ground_truth_10000.json`) 자체의 결함임.

### 4-3. 재현성 확인 — 별도 실행에서도 동일 지점에서 동일하게 재현됨

`evaluate_ocr_10000.py`로 전체(checkup + health_check)를 다시 실행한 결과(`20260609_003125_ocr_eval_10000.json`)에서도 **완전히 동일한 양상**이 확인됨.

| 구분 | 결과 |
|---|---|
| checkup_pdf 전체 평균 정확도 | 99.9% (10,000건 중 9,982건 100% — 구조적 경계 없음, 산발적 오차만 18건) |
| health_check_pdf 전체 평균 정확도 | 66.1% |
| health_check 정확도 변화 지점 | `health_check_01000.pdf` = 100% → `health_check_01001.pdf` = 33.3% (이전 실행과 **정확히 동일한 위치**에서 동일하게 끊김) |

이처럼 재실행 시에도 끊기는 위치와 양상이 한 치의 오차 없이 동일하게 재현된다는 것은, 이 현상이 메모리 누수·캐시 오염처럼 실행할 때마다 달라질 수 있는 "런타임 문제"가 아니라, **`health_check_01001.pdf`부터 시작되는 PDF 파일들 자체에 고정적으로 박혀 있는 결함**이라는 4-2의 결론을 한 번 더 뒷받침함. 또한 checkup_pdf는 같은 데이터셋·같은 파이프라인에서 99.9%를 유지해, 문제가 코드가 아니라 health_check 데이터셋 1,001번 이후 구간에 국한된 것임을 추가로 확인함.

## 5. 결론 및 보고 방향

- 수정한 두 가지 파싱 로직(키/몸무게/BMI 교차 오염 제거, 혈압 합성 레이어 중복 제거 허용 오차 조정)은 모두 의도대로 동작하며, 정상 데이터(1~1,000번)에서 **99.7%**의 정확도를 달성함 — 이것이 현재 파싱 로직의 실질 성능임.
- 10,000건 평가에서 나온 66.1%는 우리 코드의 결함이 아니라, 1,001번 이후 health_check PDF 데이터셋 자체에 "서로 다른 사람에게 동일한 값이 인쇄된" 결함이 섞여 있었기 때문에 나온 결과이며, 파싱 로직의 실제 성능을 반영하지 않음.
- 따라서 검증 결과 보고는 **"정상 구간 1,000건 기준 99.7%"**를 파싱 로직의 실질 성능으로 제시하고, 동시에 **"health_check 데이터셋 1,001번 이후 구간에 데이터 결함이 있다"**는 사실을 별도로 알려, 다른 조원이 같은 데이터로 재평가하며 시간을 낭비하지 않도록 공유하는 것이 적절함.
