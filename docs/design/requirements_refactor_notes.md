# Requirements Refactor Notes

이 문서는 코치님/조교님 피드백을 반영해 요구사항 정의서, API 명세서, NFR, 아키텍처 문서를 재정리하기 위한 작업 기준입니다.
요구사항 정의서는 사용자 관점 기능 중심으로 정리하고, NFR/아키텍처/재시도/공통 컴포넌트 같은 구현 세부사항은 별도 설계 문서로 분리합니다.

## 1. 코치님 피드백 요약

- 회원가입은 기본정보 중심으로 축소하고, 건강관리 수치와 모델 입력값은 별도 건강정보 입력 화면에서 받는다.
- 인증은 이메일 인증과 FastAPI JWT를 유지하고, 휴대폰 SMS 인증은 MVP/시연 범위에서 제외/보류한다. 소셜 로그인과 웨어러블 연동도 1차 범위에서 제외/보류한다.
- 보안 요구사항은 사용자 피해를 줄이는 방향으로 설계한다. 로그인 실패 제한은 영구 잠금보다 CAPTCHA, 일시 제한, 안내 중심으로 둔다.
- 건강정보 validation 기준과 분석 실패 시 사용자에게 보여줄 오류 범위를 명확히 정의한다.
- RAG, ML, CV, 알림, 가족, 관리자, 모니터링은 풀서비스 구현 대상으로 유지하되 세부 설계 문서를 분리한다.
- API 명세서는 REST 원칙에 맞춰 path/query parameter 기준을 정리하고, `today` 같은 특수 endpoint 남발을 피한다.
- 모델 평가는 위험군을 놓치지 않는 recall을 가장 중요하게 보고, F1/confusion matrix를 함께 본다.

## 2. 요구사항 정의서 수정 원칙

요구사항 정의서는 사용자가 실제로 수행하는 기능과 화면 흐름 중심으로 작성합니다.

- 사용자 관점의 기능 요구사항만 요구사항 정의서 본문에 둔다.
- NFR, 재시도 정책, queue, worker, 공통 컴포넌트, observability는 별도 설계 문서로 분리한다.
- 기능별 요구사항은 입력, 동작, 결과, 예외 흐름을 간결하게 적는다.
- 구현 기술명은 꼭 필요한 경우에만 보조 설명으로 둔다.
- MVP 시연용 더미/룰/mock 표현은 요구사항 정의서에서 제거하고, 구현 상태 문서에서만 구분한다.

## 3. 회원가입/인증 정책 변경

### 회원가입 입력 범위

회원가입에서는 계정 생성과 본인 기본정보에 필요한 값만 받습니다.

- 아이디
- 이메일
- 이메일 인증
- 비밀번호
- 이름
- 성별
- 생년월일
- 휴대폰 번호
- 주소: 필수 정책으로 둘 경우 기본정보에 포함
- 개인정보/민감정보 동의

키, 몸무게, 가족력, 흡연/음주/운동, 혈압, 혈당, 지질 수치 등 건강관리 지표는 회원가입에서 강제하지 않고, 가입 후 건강정보 입력 화면에서 받습니다.

### 인증 정책

- FastAPI JWT 인증을 기본 인증으로 유지한다.
- Access token과 refresh token 역할을 구분한다.
- Refresh token은 httpOnly cookie 저장 정책을 유지한다.
- 이메일 인증은 유지한다.
- 휴대폰 번호 중복확인은 유지 가능하다.
- 휴대폰 SMS 인증은 MVP/시연 범위에서 보류하고, 회원가입 필수 인증은 이메일 인증만 사용한다.
- 소셜 로그인은 1차 범위에서 제외/보류한다.
- 향후 소셜 로그인은 카카오/네이버를 우선 검토하고, 애플은 후순위로 둔다.

### 비밀번호/보안 정책

- 비밀번호 해싱은 Argon2id 도입을 검토한다.
- 로그인 실패 제한은 계정 주인 피해가 적도록 일시 제한 또는 CAPTCHA 중심으로 설계한다.
- 비밀번호 재설정 강제는 신중히 검토한다.
- 계정 잠금/제한 메시지는 내부 사유를 과도하게 노출하지 않는다.

## 7. RAG/ML/CV/챌린지/알림/가족 후속 과제

### RAG

RAG 파이프라인은 별도 설계 문서가 필요합니다. 멘토님과 아래 항목을 논의합니다.

- LangChain 또는 LangGraph 사용 여부
- embedding model 선정
- vector store 후보: pgvector, FAISS, Chroma
- 평가/관측 후보: RAGAS, LangSmith, Langfuse
- 문서 chunking 전략
- retrieval strategy
- 의료/건강 답변 안전장치
- 프롬프트 버전 관리

### ML

- x1/x2 feature schema 고정
- preprocessing pipeline 분리
- 모델 버전 관리
- threshold 관리
- `ai_inference_logs` 저장
- 위험군 recall 중심 평가
- F1, confusion matrix 병행 평가

### CV/식단

- 정확한 그램 단위 영양 계산은 후순위로 둔다.
- 초기에는 식단 기록, 이미지 기반 간편 분석, 사용자 보정 중심으로 간다.
- 영양성분 DB 매칭과 정밀 점수화는 후속 과제로 둔다.

### 챌린지

- 직접 완료 체크만으로는 이탈 가능성이 있어, 일상 기록과 자연스럽게 연결되는 체크 방식을 검토한다.
- 위험도 레벨별 챌린지 추천은 룰 기반 또는 LLM 기반으로 설계한다.
- LLM 추천 시 프롬프트 엔지니어링 구조와 안전장치를 문서화한다.
- 보상 시스템은 후속 아이디어로 둔다.

### 알림

- 알림 풀기능은 후순위다.
- 챌린지 알림과 시스템 알림은 기본 기능으로 유지한다.
- reminder schedule, notification log, 발송 worker는 후속 구현 대상으로 둔다.

### 가족공유

- 가족공유는 최후순위 또는 후속 스프린트로 둔다.
- 단, 풀서비스 범위에서는 구현 대상으로 유지한다.
- 가족 건강정보 공유는 초대/수락/권한 토글/접근 로그를 전제로 한다.

## 8. API 명세서 수정 원칙

- 목록 조회의 기준/필터/검색/기간 조건은 query parameter를 사용한다.
- 객체 식별자는 path parameter를 사용한다.
- `today`, `latest`, `summary`는 의미가 명확한 조회 endpoint로 제한하고 남발하지 않는다.
- 생성/수정/삭제는 REST method 의미에 맞춘다.
- API 명세서 템플릿은 아래 항목 중심으로 간결하게 유지한다.
  - Method
  - Path
  - 인증 필요 여부
  - Query parameter
  - Path parameter
  - Request body
  - Response body
  - 주요 오류 코드
  - 비고

## 9. 모델 평가 기준

의료/건강 위험도 모델은 위험군을 놓치지 않는 것이 중요합니다.

- 최우선 지표: recall
- 보조 지표: F1
- 오류 분석: confusion matrix
- threshold는 질환별로 분리 관리한다.
- precision만 높고 위험군 recall이 낮은 모델은 서비스 적용 전 재검토한다.
- 평가 결과는 모델 버전과 함께 기록한다.
